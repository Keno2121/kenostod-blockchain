Executive Summary

This document is a comprehensive internal security audit of the UTL Protocol MetaMask Snap (npm package: kenostod-utl-snap v2.3.0). The audit covers permission justification, network access analysis, data handling practices, code review findings, and an assessment of all potential attack vectors within the scope of the MetaMask Snaps security model.

The snap was built using @metamask/snaps-cli v6.7.0 and compiled with the Webpack bundler. All source code is publicly available at github.com/Keno2121/kenostod-blockchain.

Section 02

Snap Overview

Purpose

The UTL Protocol Snap provides MetaMask users with real-time transaction intelligence for DeFi activity on BNB Smart Chain. It is a transaction insights snap — meaning its primary purpose is to show users additional information before they sign a transaction, not to modify transactions or manage accounts.

Core Functionality

  • Fee X-Ray (onTransaction handler) — Analyzes each transaction before signing and displays MEV risk (HIGH/MEDIUM/LOW), estimated slippage loss in BNB, and LP spread cost
  • Loyalty Dashboard (RPC: utl_dashboard) — Shows staked KENO balance, rewards earned, tier status, and governance weight — all read from local snap state and public API
  • Earnings Breakdown (RPC: utl_earnings) — Fetches aggregate protocol stats from the UTL API and shows network revenue split
  • MEV Info (RPC: utl_mevInfo) — Explains MEV extraction and how UTL captures it
  • Weekly Cronjob — Runs at 9AM UTC daily. Sends weekly summary on Sundays and FAL™ opportunity alerts when profit exceeds 0.5% threshold
  • On Install — Shows a one-time welcome panel explaining all snap features

What the Snap Does NOT Do

  • Does not sign transactions on the user's behalf
  • Does not access, derive, or export private keys
  • Does not modify or redirect transaction destination addresses
  • Does not manage accounts or have endowment:keyring
  • Does not access or store seed phrases
  • Does not send user data to any third-party analytics service
  • Does not communicate with any server other than the UTL Protocol public API

Section 04

Network Access Review

All network calls originate from three functions. Each call is wrapped in a try/catch with graceful fallback to local snap state if the network is unavailable.

Endpoint 1 — Wallet Profile

GET https://utl.kenostodblockchain.com/api/utl/fee/wallet/{walletAddress}
Timeout: AbortSignal.timeout(5000)
Purpose: Read user's total toll paid and governance weight
Sends: wallet address (public on-chain identifier, not a secret)
Returns: { success, profile: { totalTollPaid }, governance: { effectiveWeight } }
Fallback: returns null → local state used unchanged

Endpoint 2 — Network Stats

GET https://utl.kenostodblockchain.com/api/utl/fee/stats
Timeout: AbortSignal.timeout(5000)
Purpose: Aggregate protocol statistics for earnings breakdown panel
Sends: nothing (public endpoint, no authentication)
Returns: { totalCollected, totalDistributed: { stakers, foundation, treasury } }
Fallback: returns null → zeros shown in panel

Endpoint 3 — FAL Opportunities

GET https://utl.kenostodblockchain.com/api/arbitrage/opportunities
Timeout: AbortSignal.timeout(4000)
Purpose: Check for high-profit arbitrage alerts in daily cronjob
Sends: nothing (public endpoint)
Returns: array of opportunity objects with profitPercent
Guard: only fires notification if profitPercent > 0.5%
Fallback: silent failure, no notification sent

Data Sent Over Network

The only user-identifying data sent over any network call is the wallet address, which is a public on-chain identifier visible to anyone who has ever received a transaction from the user. No name, email, seed phrase, private key, or any other sensitive identifier is transmitted.

TLS / HTTPS

All three endpoints use HTTPS with valid TLS certificates. No mixed-content or HTTP-only calls exist in the codebase.

Section 05

Code Review Findings

UTL-2026-001 LOW
Wallet address stored in local state without explicit user confirmation prompt
When utl_activate is called by a dApp, the user's wallet address is passed as a parameter (params.walletAddress) and stored in snap state via snap_manageState. While this is standard practice for opt-in snaps and the address is only readable within the MetaMask snap sandbox (not exposed externally), the snap does not explicitly display a confirmation dialog before storing the address.
The activation dialog (snap_dialog type: 'alert') shown during utl_activate informs the user they are opting in and explains what data will be tracked. The address is a public on-chain identifier. Severity remains Low. In v2.4.0, a type: 'confirmation' dialog will be added as a pre-step to explicit opt-in, requiring the user to click "Approve" before state is written.
UTL-2026-002 INFO
MEV risk calculation uses heuristics on contract name rather than on-chain bytecode
The mevRisk() function classifies a transaction as swap-related by checking if the to address contains known keywords ('pancake', 'swap', 'router', 'exchange') via string matching on the address. Since contract addresses are hexadecimal and do not contain these strings, this check will never match on the to address itself and effectively only triggers on value thresholds.
This is a logic limitation that results in more conservative (lower) MEV risk ratings, not a security vulnerability. Users are never shown an inflated risk. A future version will cross-reference known DEX router addresses against a maintained on-chain lookup table. Documented for transparency.
UTL-2026-003 INFO
Cronjob fires even when MetaMask is in a low-activity state
The daily cronjob schedule (0 9 * * *) has no mechanism to skip execution if the snap has not been used recently. This could cause unnecessary background API calls for users who installed the snap but are no longer active.
The cronjob contains an early return guard: if (!state.isOptedIn) return; — meaning it only makes network calls for users who have explicitly opted in. Inactive users see no impact. The API call is a single lightweight GET request with a 4-second timeout.
UTL-2026-004 INFO
No rate limiting on RPC handler calls from dApp origin
Any dApp that has the user's permission to call the snap (wallet_requestSnaps) can call RPC methods like utl_dashboard or utl_earnings in rapid succession, potentially triggering multiple API calls to the UTL Protocol endpoint.
MetaMask's snap execution model serializes RPC calls — only one can execute at a time. The UTL Protocol API is rate-limited server-side. All calls include AbortSignal timeouts. No user funds or private data are at risk from rapid calling. Documented as informational.

Areas Reviewed with No Findings

  • Private key access — No snap_getBip32Entropy, snap_getBip44Entropy, or any entropy/key derivation calls exist in the codebase. Confirmed by full source scan.
  • Transaction modification — The onTransaction handler only returns a content panel. It does not modify transaction.to, transaction.value, transaction.data, or any other field.
  • Data exfiltration — State data stored via snap_manageState is only readable within MetaMask's snap sandbox. No file system access, no WebSockets, no data sent to any server except the three documented UTL Protocol endpoints.
  • Injection attacks — All display content uses MetaMask's structured UI primitives (panel, text, heading, row, copyable) — no raw HTML rendering, no innerHTML, no XSS surface.
  • External script loading — No dynamic import(), no eval(), no remote script loading. Bundle is fully self-contained.
  • Phishing vectors — No URLs in dialogs link to lookalike or deceptive domains. All links reference utl.kenostodblockchain.com only.
  • Reentrancy / state corruption — State reads and writes are sequential (await-chained). No concurrent state mutation paths exist.

Section 08

Conclusion & Sign-Off

This internal security audit was conducted by a full review of the snap source code at utl/metamask-snap/src/index.ts (305 lines), the snap manifest (snap.manifest.json), and the compiled bundle (dist/bundle.js). All functionality, permissions, network endpoints, and data handling practices were reviewed against the MetaMask Snaps security model and the principle of least privilege.

Key conclusions:

  • Zero critical or high-severity vulnerabilities identified
  • All requested permissions are necessary and minimally scoped
  • No private key access, no transaction modification, no data exfiltration
  • Network access is limited to three read-only public endpoints with hard timeouts
  • Graceful failure on all network calls — snap functions offline
  • No raw HTML rendering — all output uses MetaMask structured UI primitives
  • The snap is safe for distribution to MetaMask users
